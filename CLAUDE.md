# CLAUDE.md

## 项目概述

NeonMe — 基于 Next.js 14 的个人展示网站。AI 公司销售总监的个人品牌站。

## 技术栈

- Next.js 14 (App Router, "use client" 全站客户端渲染)
- TypeScript (strict mode)
- Tailwind CSS 3
- Framer Motion
- lucide-react
- Jest + ts-jest (单元测试)

## 目录结构

```
app/           — 页面（前台 + 后台）
components/   — 通用组件
lib/          — 数据模型、存储、工具函数
public/       — 静态资源
```

## 核心开发者指引

# Vibe Coding Global SOP | 防AI冗余Bug标准化约束
适用项目：个人网站（前台展示 + 后台管理系统）
强制全程遵守4套开发规范，禁止违规编码。

## 1. 全局编码铁则（所有代码生成前置约束）
1. 需求模糊必须主动提问，禁止自行脑补逻辑；多方案必须罗列利弊，不能私自选定实现。
2. 代码极简原则：只实现明确需求，不额外加功能、不提前预留扩展性、不写无效异常处理；杜绝过度设计。
3. 修改存量代码仅改动需求相关行，不重构、美化无关代码；对齐项目原有风格；只清理本次改动产生的无用导入/变量。
4. 复杂任务先输出「分步计划+每步验收标准」，确认后再写代码。

## 2. 新功能开发：强制TDD模式
时机：编写业务代码之前
1. 梳理需求、入参、边界场景
2. 调用 /anthropic-skills:test-helper 生成对应单元/组件测试
3. 先看到测试失败（红），再写最小代码让测试通过（绿）
4. 仅在不改变逻辑前提下精简重构
测试优先级：
1. 最高：lib/store.ts、lib/useDragSort.ts
2. 中等：ImageUpload、Toast 组件
3. 最低：页面展示层

## 3. Bug/报错修复：系统化调试流程
时机：准备修改代码修复问题之前
1. 复现：固定操作路径，100%稳定复现问题
2. 写测试：生成可触发bug的测试用例，确认用例失败
3. 定位：分析堆栈、参数，输出根因，拒绝表面补丁
4. 修复：最小改动解决根源问题
5. 验证：测试通过+人工复测，确认无新问题引入

## 4. 提交/合并前强制自检
时机：任务完成、大功能收尾、git merge / 发起Code Review之前
逐项校验通过才可收尾：
- [ ] 需求全部实现，无缺漏、无多余功能
- [ ] 对应测试全部运行通过
- [ ] 核心流程手动自测正常（增删改查、上传、拖拽、页面跳转）
- [ ] 控制台、接口无 400/500、类型、语法报错
- [ ] 所有代码修改均可追溯本次需求
- [ ] 空值、非法入参等边界场景已处理

## 5. 存量Bug整改规则
1. 分级处理：P0阻塞问题优先，其次P1功能异常，再处理体验瑕疵，最后清理技术债
2. 一次只修复一个Bug，逐个闭环，禁止一次性大范围批量修改重构
3. 每修复一个Bug，新增对应测试用例防止反复复发

## 6. 长期避坑规则
1. 任务必须拆成微型步骤开发，禁止一次性生成整套大模块代码
2. 配合项目ESLint + TS类型校验，即时拦截低级语法/类型错误
3. 你负责架构、需求、优先级、最终验收；你只负责落地编码、写测试、定向改bug
4. 每3~5轮迭代，统一清理低优先级遗留问题、补充测试覆盖率
